31 ## Introduction
32 
33 This is a lisp interpreter written largely in [ANSI C][], with some [c99][], it is a
34 non-standard variant of lisp.
35 
36 <div id='Design goals'/>
37 ## Design goals
38 
39 A summary of the design goals:
40 
41 * Simple and Small (~ 5000 lines of code)
42 * Extensible
43 * No implicit state
44 * Used as a generic utility library
45 
46 The following functionality would be nice to put in to the core interpreter;
47 Support for matrices and complex numbers (c99 \_Complex), UTF-8 support, text
48 processing (diff, grep, sed, à la perl), map and looping constructs
49 from scheme, doc-strings, optional parsing of strings and float, apply,
50 proper constants and a prolog engine.
51 
52 But this is more of a wish list than anything. Complex number support has
53 been added in a branch of the [git][] repository. Another interesting
54 possibility would be to use a big memory pool and allocate objects from that,
55 pointers would be replaces with indexes into this table, it would make saving
56 the state of the interpreter and restoring it a lot easier, plus would allow
57 for more advanced memory management in a portable way.
58 
59 There are a lot of things that need tidying up and rethinking about,
60 especially when it comes to the interface of the library and the naming of
61 functions. More comments are also in order in the code explaining what the code
62 is meant to achieve is also in need.
63 
64 ### Rationale
65 
66 * Simple and Small (~ 5000 lines of code)
67 
68 The interpreter and the language should be simple to understand. It provides
69 more of a DIY lisp environment where most facilities are not provided but are
70 up to the users to implement. The interpreter is currently slightly over 5000
71 lines in of C formatted with "GNU indent", and under it with the way I like the
72 code formatted.
73 
74 The count does not include the modules, experimental code or the unit tests in
75 [unit.c][] and these are not part of the core interpreter, these should grow as
76 large as possible!
77 
78 * Extensible
79 
80 The core language is very small, with few primitive operations, as such it
81 needs to be extensible. New primitive functions needed be added by library
82 users, code fragments evaluated and even new types need adding.
83 
84 * No implicit state
85 
86 It should be possible to run multiple instances of the interpreter in the same
87 address space, this requires no implicit or hidden state in the interpreter
88 design, which impedes multiple instances running at once. There is currently is
89 hidden state which is needed by the module system to function; there is a
90 single global lisp environment that each module adds its code to when it is
91 loaded.
92 
93 If the module system is not used then multiple lisp interpreters can be run in
94 different threads. This could be remedied if instead of the modules having
95 constructors and destructors that add their functions into a global environment
96 the lisp environment acquired a function with "dlsym", which would have to be
97 part of the module specification and have a fixed name, and passed in the lisp
98 environment it was operating on to that function.
99 
100 * Used as a generic utility library
101 
102 The library includes routines that might be useful in other programs that it
103 could have chosen not to expose, mostly relating to I/O and strings.
104 
105 * Portable
106 
107 The core library is written in C99, as such it should compile anywhere. Only a
108 very small wrapper around the library is needed to make a working interpreter.

119 <div id='Building'/>
120 ## Building
121 
122 To build the interpreter a C compiler for a hosted platform is needed. It has
123 no other dependencies apart from the C library. A makefile is provided, but the
124 following commands should all work for their respective compilers:
125 
126  gcc *.c -o lisp
127  tcc *.c -o lisp
128  clang *.c -o lisp
129 
130 The preprocessor defines that add or change functionality are:
131 
132  USE_DL:
133  Add the dynamic loader interface
134  NDEBUG:
135  The interpreter liberally uses assertions, this
136  disables them.
137  USE_INITRC:
138  Add functionality to read from an initialization
139  file ("~/.lisprc" or "%HOMEPATH%\.lisprc").
140  USE_ABORT_HANDLER
141  This adds a handler for the SIGABRT, which is
142  raised by assert(). It adds a signal handler
143  which catches the signal, prints out a stack
144  trace using backtrace from "execinfo.h" and then
145  continues with the abort.
146 
147 Using the makefile instead (either "make" or "make all") will build the "lisp"
148 executable as well as the "liblisp.a" library. It will also build "liblisp.so"
149 The libraries will be built with debugging symbols.
150 
151 The line editing library is limited to Unix systems which support a
152 [VT220][] terminal emulator (or are lucky enough to the actual terminal
153 hooked up!), which should be all of them. It may also work with POSIX
154 compatibility layers such as [Cygwin][], but this has not been tested. The
155 repository is set up a fork of the [linenoise][] library that is simply called
156 [libline][], this has limited support for ["Vi"][] like key bindings. This is
157 compiled as a module and will only be loaded if compiled the lisp interpreter
158 has been compiled with "USE\_DL", and if the line editing library is available
159 on your platform.
160 
161 Color support can be optionally enabled on all output channels and is reliant
162 on [ANSI escape codes][] on the terminal being displayed correctly. If it is
163 not simply do not enable color, it is only a minor feature.

165 <div id='Deviations from other lisps'/>
166 ## Deviations from other lisps
167 
168 * Case sensitivity
169 
170 Case sensitivity is both easier to implement, more general and allows
171 information to be encoded in case, much like in language with acronyms.
172 If the implementation were to be case insensitive the question of what
173 character encoding and sets we use would come up, we might design a lisp which
174 is case insensitive with the Latin alphabet, but why not with the Cyrillic
175 alphabet also?
176 
177 * Does not follow any standards
178 
179 As this implementation does not follow any standards there are bound to be
180 general behaviors and corner cases that are out of line with other standards
181 implementation, or are combinations of behaviors from other standards. The
182 implementation is the standard for now, which is a downside.
183 
184 * '#' as comments as well as ';'
185 
186 Another major deviation is that comments are represented by '#' as well as ';'
187 so that source code can easily be run as a Unix script with no special syntax
188 in the usual manner.
189 
190 * Subtly different functions
191 
192 Functions may behave differently from those with the same name defined in
193 another language.
194 
195 * F-Expressions
196 
197 As they were easy to add, F-expressions were added.

241 ### Design Faults
242 
243 * Using the C stack
244 
245 Instead of using the C stack for the interpreter stack it would have been much
246 better to use a heap allocated stack, this would mean closure could be
247 implemented easily and portably. Not only this but the stack could grow without
248 worrying about overflow.
249 
250 * Making a pure interpreter instead of a register machine
251 
252 A pure interpreter is easy to implement, but limited, it would be better to
253 make a lisp interpreter based around a register machine built for executing
254 lisp.
255 
256 * Using pointers
257 
258 If indexes into arrays (or data and function pointers) it would be easier to
259 store program state as an image, which is a nice feature. It would mean that
260 memory usage could be brought down a lot in a *portable* manner.
261 
262 * F-Expressions
263 
264 F-Expressions are very easy to implement, but very difficult to use, a proper
265 macro system should be implemented instead of relying on F-Expressions.
266 
267 ### Testing and bug mitigation
268 
269 There are a few methods used (or aspired to) in this interpreter when it comes
270 to mitigating bugs, documentation, profiling the code, and hopefully catching
271 bugs introduced due to refactoring.
272 
273 * Unit Tests
274 
275 There are two parts to the unit test frame work, a C program that calls
276 functions directly out of the library and unit test code written in lisp. The
277 aim of the C program is to test all of the functions that can reasonably be
278 tested by it (the higher level REPL functions will have to be tested by the
279 lisp code and the user).
280 
281 * Valgrind
282 
283 Valgrind is used regularly to catch memory leaks. Currently the interpreter
284 does not contain any known memory leaks, however some modules do leak memory,
285 it is unclear if the modules themselves of the libraries they link against are
286 responsible for the leaks.
287 
288 * Doxygen
289 
290 Doxygen is used to document most of the function interfaces (and all public
291 ones), create a "to-do" list, a list of known bugs, make notes and provide
292 a list of warnings to the programmer. It generates a list of warnings, which is
293 logged to "doxygen.log".
294 
295 * Gprof
296 
297 "gprof" has been used and has helped find and remedy the highest offenders in
298 terms of sloth (assoc).
299 
300 * Gcov
301 
302 "gcov" has not been used at the moment, either for profiling or coverage tests,
303 this could be used in conjunction with the unit test framework to make sure it
304 covers a high percentage of the code base.
305 
306 * Splint
307 
308 Using splint (3.1.2) has been unsuccessful, it fails to parse most of the C
309 files correctly and so cannot produce any warnings. The "-weak" setting would
310 be used as there is no intention to mark up the source code with splint type
311 information.
312 
313 * Asserts
314 
315 Asserts are used throughout the code base, in fact disabling them gives about a
316 10-20% speed boost. When changes are made, asserts have prevented subtle
317 corruption and provided hints as to where the problem has occurred, failing
318 fast and quickly instead of giving the impression that a change might have been
319 successful.

387 ### [Lisp][]
388 
389 [Lisp][] is the second oldest language after [FORTRAN][] that still has
390 descendants still in use. Despite its age it is a very high level language
391 built on simple principles; it had features that are now finding their way
392 into newer programming languages such as the [lambda procedure][]. It is a
393 multi-paradigm programming language built around a few core concepts:
394 
395 1. [S-Expressions][]
396 2. [Garbage Collection][]
397 3. A mostly functional style of programming.
398 
399 And their are various minor attributes that could also be said about the
400 language as well.
401 
402 The first point however is very important, in [Lisp][] code and data are
403 represented in the same way, by the [S-Expression][]. Naturally any
404 programming language is going to have primitives and procedures for handling
405 the data types it is built to work with; [MATLAB][] is good at handling matrices,
406 [AWK][] with its associative arrays and regular expressions is good with
407 strings, [Lisp][] is good at handling [S-Expressions][].
408 
409 Given the statements that:
410 
411 1. In [Lisp][] code and data have the same format, [S-Expressions][].
412 2. [Lisp][] is designed to process [S-Expressions][].
413 
414 It follows that [Lisp][] must be good at, or at least able to; read in,
415 manipulate, generate and write out [Lisp][] programs with relative ease. It can
416 also evaluate these expressions at run time. To put it another way; it is
417 possible to write [Lisp][] programs that write [Lisp][] programs.
418 
419 The second important point, [Garbage Collection][], means that the programmer
420 does not have to worry about how objects are allocated or cleaned up, but it
421 does come at a price. The collector used here is a simple stop-the-world
422 [mark and sweep][] collector, this may add pauses at seemingly random times
423 throughout the programs execution.
424 
425 The third point mostly is the least important, but means that techniques such
426 as [recursion][] (with [tail call][] optimization in this implementation) are
427 preferred and functions with [side effects][] are discouraged.
428 
429 Some of these points are more historically important, most new languages now
430 have garbage collection. Lisp also was one of the first languages to have a
431 [REPL][], again many new languages also have this feature. REPL stands for
432 Read-Evaluate-Print-Loop and this is what the interpreter does, it reads in a
433 lisp expression, evaluates it, prints the result and loops to the beginning to
434 read in a new expression. This gives instant feedback on whether an operation
435 worked or now.

482 ## An introduction to the interpreter
483 
484 The lisp interpreter is a thin wrapper around the liblisp library, it adds
485 functionality to load compiled modules into the running executable image and then
486 runs the interpreter. What it does after this is controlled by by the arguments
487 passed into the lisp library program. As much functionality is the same
488 regardless of what system you are running the interpreter on, this cannot be
489 said to be the same case for the loaded in modules which might exist only on a
490 subset of the support platforms.

493 ### Command line arguments
494 
495 The options that can be passed into the lisp interpreter are as follows (taken
496 from the man-pages):
497 
498  * -h
499 
500 Print out the usage string and exit successfully.
501 
502  * -c
503 Turn on colorization of printed out S-Expressions, it assumes your terminal can
504 handle ANSI color codes. If it cannot do not select this option. It does no
505 checking for whether it is writing to terminal or not with isatty(3).
506 
507  * -p
508 
509 This will turn the prompt on, by default no prompt is given when typing so as
510 to allow output to be piped to other programs following the "No news is good
511 news" Unix maxim.
512 
513  * -E
514 
515 This will turn on the line editor when reading from stdin.
516 
517  * -H
518 
519 If an error is encountered the interpreter will be halted rather than trying to
520 recover.
521 
522  * -i file
523 
524 This option allows the user to supply a file name that would otherwise look
525 like an argument to the lisp interpreter, for example if the file name began
526 with a "-" character.
527 
528  * -- file
529 Same as "-i".
530 
531  * -e string
532 Evaluate a string as input, this disables reading from stdin(3).
533 
534  * -o file
535 Open "file" and redirect output to it.
536 
537  * file
538 An argument that is not preceded by a valid option that takes an argument is
539 treated as a file to be read from. Reading from a single file disables reading
540 from stdin(3).
541 
542  * -
543 
544 This forces reading from stdin(3). By default the interpreter will read from
545 stdin(3) but this behavior is disabled if an argument has been evaluated with
546 "-e" or a file has been read in.
547 
548 If no file is given the interpreter reads input from stdin(3), if one is given
549 then the interpreter will exit after having read all input files or evaluated
550 all strings.
551 
552 Arguments are read in order and executed as soon as they are encountered.
553 
554 As an example:
555 
556  $ ./lisp -Epc init.lsp -
557 
558 Will run the lisp interpreter, turn on the line editor (if available), turn the
559 printing out the prompt on, and colorize the output to standard out. It will
560 then read in and evaluate all the expressions in the file "init.lsp", after it
561 has done that it will start an interactive session with the user.

623 ### Atoms
624 
625 An atom is the simplest data structure available to the lisp environment, it is
626 a primitive data structure; lists are composed of collections of atoms and
627 other lists and an atom cannot further be divided into smaller lists or atoms.
628 
629 Atoms usually evaluate to themselves, apart from symbols which evaluate to
630 another atom or expression depending on the environment.
631 
632 For example:
633 
634  > 2.0 # Floating point numbers are self evaluating
635  2.0
636  > 2 # Integer numbers are self evaluating
637  2
638  > +
639  <SUBR:4231056> # Evaluates to an internal subroutine
640  > '+
641  (quote +) # quote postpones evaluation
642  > "Hello, World!" # strings are self evaluating
643  "Hello, World!"
644 
645 Atoms can have different types, such as a hash, an IO port, a number, a string,
646 and a floating point number. They will be colorized differently depending on
647 what they are.

649 ### CONS cells
650 
651 The [cons cell][] is the basic data structure from which lists and trees, and
652 so most structured data in this lisp apart from [associative arrays][], are
653 built. The *cons* subroutine is used to allocate and build new cons cells and
654 lists.
655 
656 A cons cell consists of two cells, each of which can hold either a pointer to
657 other cons cells or data. Each cell will contain a few bits to determine what
658 the data is, or whether it is a pointer, as well as for other purposes such as
659 garbage collection.
660 
661 The proper list:
662 
663  (1 2 3)
664 
665 Looks like this in memory:
666 
667  CAR CDR
668  .-----.-----. .-----.-----. .-----.-----.
669  | 1 | PTR |---->| 2 | PTR |---->| 3 | NIL |
670  .-----.-----. .-----.-----. .-----.-----.
671 
672 The first cell in the pair is called the "car" cell and the second the "cdr"
673 cell for historical reasons. "PTR" represents a pointer to another cell and
674 "NIL" represents the "nil" symbol, which is used to terminate a proper list.
675 
676 Lists which are not terminated by a "nil" can also be made, they are called
677 [dotted pairs][].
678 
679 The [dotted pair][]:
680 
681  (1 . 2)
682 
683 Looks like this:
684 
685  .-----.-----.
686  | 1 | 2 |

1521 
1522 Coerce one type into another, if the coercion is possible, if not it throws an
1523 error. All objects can be coerced into the same type as the object, for example
1524 hashes can be coerced into hashes, integers into integers, etc.
1525

1528 "-1", "0", "99" and not "99a", "not-a-number" or even "1.0".
1529

1531

1535

1537 forms that would be valid integer sequences if feed into the interpreter raw.
1538

1540

1546

2308 ### dynamic loader
2309 
2310 This is the interface to the dynamic module loader. This can be used to load
2311 compiled modules at run time so the interpreter can access compiled subroutines
2312 after the interpreter itself has been compiled. This interface does not
2313 implement a way to close loaded modules as there is no way to invalidate
2314 previously acquired subroutines.
2315 
2316 * dynamic-open
2317 
2318 Open up a module for use, it returns a user defined type representing the
2319 shared library handle that can be used with *dynamic-symbol*.
2320 
2321 * dynamic-symbol
2322 
2323 Get a symbol from a compiled module, the symbol should be a function abiding by
2324 the function type of all the built in subroutines. The following C typedef is
2325 how the returned subroutine should look like:
2326 
2327  typedef cell *(*subr)(lisp *, cell*);
2328 
2329 This function will return 'error if the subroutine could not be found (or there
2330 was another error, which one can be found with dynamic-error).
2331 
2332 Below is an example, given a shared module "libexample.so", with the example
2333 code:
2334 
2335  #include <liblisp.h>
2336  cell* subr_example(lisp *l, cell *args) {
2337  if(!cklen(args, 1))
2338  RECOVER(l, "\"expected (int)\" '%S", args);
2339  return mkint(l, intval(car(args)) * 2);
2340  }
2341 
2342 Make sure ldconfig has been run after installing libexample.so into a directory
2343 that the loader can find it, the following code in the interpreter will load
2344 that shared library and allow the function to be run.
2345 
2346  # (dynamic-symbol DYNAMIC-MODULE STRING)
2347  > (define a-module (dynamic-open "libexample.so"))
2348  <DYNAMIC-MODULE:34437120>
2349  > (define example-subr (dynamic-symbol a-module "subr_example"))
2350  <SUBR:47982636828560>
2351  > (example-subr 4)
2352  8
2353  > (example-subr -4)
2354  -8
2355 
2356 * dynamic-error
2357 
2358 This function returns an error string representing the latest error to occur
2359 (if any) resulting from an error with the dynamic loader.
2360 
2361  > (define a-module (dynamic-open "libexample.so"))
2362  <DYNAMIC-MODULE:34437120>
2363  > (dynamic-symbol a-module "subr_does_not_exist")
2364  error
2365  > (dynamic-error)
2366  "libdoesnotexist.so: cannot open shared object file: No such file or directory"
